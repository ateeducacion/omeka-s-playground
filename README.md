# Omeka S Playground

Browser-based Omeka S playground for static hosting. The app is split into a shell (`index.html`) and a runtime host (`remote.html`), then uses a Service Worker plus a dedicated `php-cgi-wasm` worker to serve Omeka requests from a readonly Omeka bundle with a writable browser-persistent overlay.

The Omeka application source is built from `ateeducacion/omeka-s` branch `feature/experimental-sqlite-support`. The bundle/build pipeline is adapted from the inspected `erseco/moodle-playground` reference: build a manifest, build a VFS image, keep the core readonly, and persist only mutable paths.

## Architecture

- `index.html`: shell UI with toolbar, address bar, runtime selector, reset, export/import, log panel, and iframe viewport.
- `remote.html`: runtime host with no shell chrome. It registers the Service Worker and loads the embedded Omeka iframe for one isolated browser scope.
- `sw.js`: intercepts scope-tagged requests and forwards them to the dedicated PHP worker.
- `php-worker.js`: boots `php-cgi-wasm`, mounts the Omeka bundle, writes SQLite/local config files, runs the Omeka installer tasks programmatically, and answers HTTP requests.
- `assets/manifests/latest.json`: describes the current Omeka bundle version and VFS image.
- `assets/omeka/*.vfs.*`: readonly Omeka browser bundle generated by `make bundle`.

## Runtime Model

Runtime definitions live in `playground.config.json`. v1 ships one working runtime entry:

- `php83-cgi`

The runtime selector is wired to those entries. Switching runtime reloads the remote host and causes the PHP worker to reboot with the selected runtime definition. To add a new runtime, add a new entry in `playground.config.json` and ensure `scripts/sync-browser-deps.mjs` vendors the matching runtime assets and extensions.

## Storage Model

The design is:

- readonly core bundle in `assets/omeka/*.vfs.bin` + `*.vfs.index.json`
- writable overlay persisted in the browser for:
  - SQLite database
  - `config/database.ini`
  - `config/local.config.php`
  - `files/`
  - session/log/temp state

Each tab/session gets a `scopeId` so multiple tabs do not share the same worker channels or shell session state.

## Automatic Omeka Install

The browser runtime does not require the user to fill the Omeka installer form.

At first boot the PHP worker:

1. mounts the readonly Omeka core bundle;
2. writes `config/database.ini` for SQLite;
3. writes `config/local.config.php` overrides;
4. invokes Omeka installer tasks programmatically;
5. creates the default global admin user from `playground.config.json`;
6. stores manifest state so subsequent reloads skip reinstall unless the bundle version changes and reset-on-mismatch is enabled.

Default credentials are configured in `playground.config.json`.

## Local Development

Prerequisites:

- Node.js + npm
- Composer
- Python 3
- `git`

Commands:

```bash
make up
```

Or step by step:

```bash
make prepare
make bundle
make serve
```

Default local URL:

```text
http://localhost:8080/
```

## Make Targets

- `make up`: install deps, vendor browser runtime assets, build the Omeka bundle, and serve locally
- `make prepare`: install npm deps and prepare runtime/vendor directories
- `make bundle`: fetch Omeka, run Composer install, build the VFS image, and generate the manifest
- `make serve`: start a static local server
- `make clean`: remove generated bundle and vendored runtime assets
- `make reset`: alias for cleaning generated state/caches in this repo

## GitHub Pages Deployment

The project is deployable as a static site. The Pages workflow:

1. installs npm dependencies;
2. vendors browser runtime assets;
3. fetches/builds the Omeka bundle;
4. uploads the site as a Pages artifact;
5. deploys from the repository subpath.

`.nojekyll` is included so vendored/runtime files are not filtered by Pages.

    "email": "admin@example.test"
  }

## Resetting Saved State

- Use the shell Reset button to force a clean remote reload for the current scope.
- If the bundle version changes and `resetOnVersionMismatch` is `true`, the runtime will recreate the mutable overlay automatically.

## Limitations

- The code ships one runtime entry in v1. The selector is real, but only one runtime is bundled by default.
- Browser persistence/import-export is currently shell-session oriented; a full overlay snapshot/export path still needs hardening.
- Omeka needs PHP extensions beyond base CGI. The build expects the vendored `php-wasm-*` extension packages listed in `package.json`.
- Safari and Firefox need real browser validation for IndexedDB-backed persistence and Service Worker behavior.
- The readonly VFS mount path is implemented, but performance and mutability boundaries should be hardened with real end-to-end runtime validation against the generated bundle.

## Browser Compatibility

Target baseline:

- Chromium current
- Firefox current
- Safari current desktop, with expected extra validation around IndexedDB and worker behavior
